# `render.sh` at a Narrow Width Is NOT a True Mobile Viewport — Judge Mobile Overflow via CDP

date: 2026-06-19
scope: frontend / verification / tooling
rule: render.sh at a 390px width is NOT a true mobile viewport (runs at DSF=2); judge mobile overflow via CDP device emulation instead.

## Why
Validating the new `/prototype` skill on an onboarding mock, I rendered each screen with `.claude/skills/diagram/render.sh "$F?screen=plan" out.png 390 1300` to check mobile and SAW clear horizontal overflow — plan cards, the 3-step indicator and the "MOST POPULAR" badge all clipped at the right edge. I trusted the pixels (the skill's own cardinal rule) and called it a real defect. The fresh skeptic refuted it: at a **true** 390-CSS-px viewport driven by CDP (`Emulation.setDeviceMetricsOverride`, `mobile:true`), `document.documentElement.scrollWidth === clientWidth === 390` on all six screens, the `@media (max-width:640px)` rules matched, the step indicator wrapped to two rows, and everything fit. The clipping was a **render.sh artifact**: `render.sh` runs Chrome with `--force-device-scale-factor=2` and sizes the OS window, so a "390" request did not produce a 390-CSS-px layout viewport — the mobile breakpoint never fired and it screenshotted a **desktop** layout cropped into a narrow frame. So a code-read-beats-pixels tool can itself lie about pixels when the tool doesn't model the thing you're testing (here, the CSS viewport). Note the `<meta name="viewport" content="width=device-width,initial-scale=1">` WAS present — without it a real phone would also fall back to a ~980px desktop layout, so that meta tag is the other half of "actually responsive."

## How to apply
`render.sh` is correct for **desktop** per-screen screenshots (`?screen=ID` at 1440-ish) — keep using it for the look loop. But do **not** judge mobile overflow / responsive reflow from a narrow `render.sh` window; it gives false positives (clipped desktop) and can give false negatives. To verify mobile, drive a real browser via the committed CDP harness (`.claude/skills/sprite-anim/verify-anim.mjs`) with `Emulation.setDeviceMetricsOverride({width:390,height:844,deviceScaleFactor:2,mobile:true})`, then compare `documentElement.scrollWidth` vs `clientWidth` (equal = no overflow) and screenshot. Always keep a `width=device-width` viewport meta in the file or the breakpoints are moot on real devices. When a render disagrees with a CDP/device measurement, the device measurement wins for "does this overflow on a phone."
